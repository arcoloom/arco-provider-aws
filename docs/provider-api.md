# Provider Usage and API Reference

This document describes how the provider in this repository is invoked by a host application. Although the current example binary is named `arco-provider-aws`, this guide is intentionally written from the perspective of a generic cloud provider. The focus is the process model, transport protocol, authentication, RPC contract, and field semantics rather than any single cloud vendor.

Protocol sources:

- Entry point: `cmd/arco-provider-aws/main.go`
- Runtime handshake: `internal/runtime/server.go`
- gRPC authentication: `internal/transport/grpc/auth.go`
- Service definition: `proto/arco/provider/v1/provider.proto`

## 1. Overview

The provider runs as a standalone process and exposes an authenticated gRPC service named `ProviderService`. The typical host-side flow is:

1. Start the provider binary.
2. Read the startup JSON from the first line of `stdout`.
3. Extract `address` and `token`, then establish a gRPC connection.
4. Attach `authorization: Bearer <token>` to every RPC.
5. Call `GetProviderInfo` first, then invoke connection validation, catalog, pricing, or instance lifecycle RPCs as needed.
6. Close the gRPC connection and stop the provider process when finished.

The current gRPC service definition is:

```proto
service ProviderService {
  rpc GetProviderInfo(GetProviderInfoRequest) returns (GetProviderInfoResponse);
  rpc GetProviderSchema(GetProviderSchemaRequest) returns (GetProviderSchemaResponse);
  rpc ValidateConnection(ValidateConnectionRequest) returns (ValidateConnectionResponse);
  rpc Ping(PingRequest) returns (PingResponse);
  rpc ListRegions(ListRegionsRequest) returns (ListRegionsResponse);
  rpc ListAvailabilityZones(ListAvailabilityZonesRequest) returns (ListAvailabilityZonesResponse);
  rpc GetSpotData(GetSpotDataRequest) returns (GetSpotDataResponse);
  rpc StartInstance(StartInstanceRequest) returns (StartInstanceResponse);
  rpc StopInstance(StopInstanceRequest) returns (StopInstanceResponse);
  rpc ListActiveInstances(ListActiveInstancesRequest) returns (ListActiveInstancesResponse);
  rpc ListInstanceTypes(ListInstanceTypesRequest) returns (ListInstanceTypesResponse);
  rpc GetInstanceTypeInfo(GetInstanceTypeInfoRequest) returns (GetInstanceTypeInfoResponse);
  rpc GetInstancePrices(GetInstancePricesRequest) returns (GetInstancePricesResponse);
}
```

## 2. Runtime and Authentication

### 2.1 Startup Handshake

When the provider starts, it listens on a local TCP address and writes one JSON object to `stdout`. The host must read that line before sending any gRPC requests.

Example:

```json
{
  "protocol": "grpc",
  "address": "127.0.0.1:43127",
  "port": 43127,
  "token": "c5f89e...<redacted>...",
  "pid": 12345
}
```

Field reference:

| Field | Type | Description |
| --- | --- | --- |
| `protocol` | `string` | Currently fixed to `grpc`. |
| `address` | `string` | Address used by the host for the gRPC connection. |
| `port` | `int` | Listening port, useful for logs, debugging, or health checks. |
| `token` | `string` | Session-scoped Bearer token for this provider process. |
| `pid` | `int` | Provider process ID. |

Rules:

- The first line of `stdout` must remain reserved for the startup JSON.
- Runtime logs should go to `stderr` so the handshake output stays clean.
- If the host cannot read a valid startup JSON payload, the provider should be treated as failed to start.

### 2.2 gRPC Authentication

All RPCs are guarded by a unary interceptor that validates the session token. The caller must send the following metadata:

```text
authorization: Bearer <startup-info.token>
```

If the token is missing, malformed, or does not match, the provider returns `Unauthenticated`.

### 2.3 Recommended Call Order

| Order | RPC | Purpose |
| --- | --- | --- |
| 1 | `GetProviderInfo` | Discover provider identity, resource planes, supported services, and auth schemes. |
| 2 | `ValidateConnection` | Check whether credentials and scope are accepted before resource operations. |
| 3 | `Ping` | Optional liveness or round-trip check. |
| 4 | Business RPCs | Invoke catalog, pricing, spot/preemptible, or lifecycle operations as needed. |
| 5 | Stop process | Release provider resources. |

## 3. Shared Data Structures

### 3.1 `RequestContext`

Any request that should participate in tracing can carry `RequestContext`:

| Field | Type | Description |
| --- | --- | --- |
| `request_id` | `string` | Request identifier, ideally generated by the host. |
| `caller` | `string` | Caller name such as controller, scheduler, or CLI. |
| `trace_id` | `string` | Cross-service trace identifier. |

Recommendations:

- Use `request_id` for idempotency analysis and log correlation.
- Use `trace_id` to include provider requests in a wider tracing pipeline.

### 3.2 `ConnectionScope`

`ConnectionScope` describes the environment in which the request should execute:

| Field | Type | Description |
| --- | --- | --- |
| `account_id` | `string` | Account, tenant, or project scope identifier. |
| `region` | `string` | Logical region. Some RPCs use it as the default region. |
| `endpoint` | `string` | Custom API endpoint. |
| `endpoint_region` | `string` | Signing or routing region to use with a custom endpoint. |
| `attributes` | `map<string,string>` | Extension attributes for provider-specific context. |

Design notes:

- The host should treat `scope` as connection environment, not only geography.
- Private endpoints, alternate API gateways, and multi-tenant routing hints can be placed here.

### 3.3 `Credentials`

`Credentials` uses a protobuf `oneof`, so exactly one authentication shape should be populated per request. The host should inspect `GetProviderInfoResponse.metadata.supported_auth_schemes` first, then fill the matching credential block.

The protocol currently includes these built-in credential messages:

#### `AwsIamCredentials`

| Field | Type | Description |
| --- | --- | --- |
| `access_key_id` | `string` | Access identifier. |
| `secret_access_key` | `string` | Secret key. |
| `session_token` | `string` | Temporary session token. |
| `role_arn` | `string` | Role identifier. |
| `external_id` | `string` | External identity value. |

#### `AzureClientSecretCredentials`

| Field | Type | Description |
| --- | --- | --- |
| `tenant_id` | `string` | Tenant identifier. |
| `client_id` | `string` | Client identifier. |
| `client_secret` | `string` | Client secret. |
| `subscription_id` | `string` | Subscription or project scope identifier. |

#### `GcpServiceAccountCredentials`

| Field | Type | Description |
| --- | --- | --- |
| `project_id` | `string` | Project identifier. |
| `client_email` | `string` | Service account email. |
| `private_key` | `string` | Private key material. |
| `private_key_id` | `string` | Private key ID. |

Generic meaning:

- The protocol expresses pluggable authentication shapes; it does not imply that every provider supports every scheme.
- A provider that supports only one auth scheme should declare that explicitly in `supported_auth_schemes`.
- If a new auth model is needed, extending the `Credentials` `oneof` is the preferred approach.

### 3.4 `Warning`

Many responses include `repeated Warning warnings`:

| Field | Type | Description |
| --- | --- | --- |
| `code` | `string` | Machine-readable warning code. |
| `message` | `string` | Human-readable message for logs or UI. |

Contract:

- `warnings` are not the same as RPC failures.
- A provider can return warnings when the request succeeds but results are partial, degraded, or only partially matched.
- Hosts should handle both transport errors and successful responses with warnings.

## 4. API Reference

All field names below map directly to `proto/arco/provider/v1/provider.proto`. If you use `grpcurl` or another protobuf JSON client, send request payloads using protobuf JSON field names.

### 4.1 `GetProviderInfo`

Purpose: Return provider metadata and declared resource planes.

Request: `GetProviderInfoRequest`

No fields.

Response: `GetProviderInfoResponse`

| Field | Type | Description |
| --- | --- | --- |
| `metadata.name` | `string` | Provider name. |
| `metadata.version` | `string` | Provider version. |
| `metadata.cloud` | `Cloud` | Cloud type targeted by the provider. |
| `metadata.supported_auth_schemes` | `repeated AuthScheme` | Supported authentication schemes. |
| `metadata.resource_planes` | `repeated ResourcePlane` | Resource planes owned by this provider, such as `COMPUTE`, `STORAGE`, or `NETWORK`. |
| `metadata.supported_services` | `repeated string` | Supported service domains such as `compute`, `catalog`, or `pricing`. |
| `metadata.capabilities` | `map<string,string>` | Extension capability map, suitable for feature flags. |

Host guidance:

- Call this once right after startup and cache it for the lifetime of the process.
- Use `supported_auth_schemes` to decide how to construct `Credentials`.
- Use `resource_planes` as the primary host-side routing signal instead of hard-coding provider names.
- Use `supported_services` and `capabilities` for finer-grained feature gating within a resource plane.

### 4.2 `ValidateConnection`

Purpose: Validate whether credentials and connection scope are accepted by the provider.

Request: `ValidateConnectionRequest`

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `context` | `RequestContext` | No | Request context. |
| `credentials` | `Credentials` | Yes | Connection credentials. |
| `scope` | `ConnectionScope` | No | Connection scope. |
| `options` | `map<string,string>` | No | Provider-specific extension options. |

Response: `ValidateConnectionResponse`

| Field | Type | Description |
| --- | --- | --- |
| `accepted` | `bool` | Whether the provider accepts the supplied connection parameters. |
| `message` | `string` | Result summary. |
| `warnings` | `repeated Warning` | Non-blocking warnings. |

Call semantics:

- `accepted=false` means business-level validation failed; it does not necessarily mean the RPC itself failed.
- Hosts can use this as a preflight check, but should not assume every provider performs a full connectivity test.
- The current sample implementation validates STS caller identity, optional account scoping, and EC2 regional read access.
- If `DescribeAvailabilityZones` is denied, the current sample may still accept the connection and return a `REGION_VALIDATION_SKIPPED` warning.

| --- | --- | --- |

### 4.7 `StartInstance`

Purpose: Create or start a compute instance.

| `context` | `RequestContext` | No | Request context. |
| `credentials` | `Credentials` | Yes | Credentials required to create resources. |
| `scope` | `ConnectionScope` | No | Default region, endpoint, and execution context. |

| `user_data` | `string` | No | Bootstrap script or initialization data. |
| `options` | `map<string,string>` | No | Generic request options such as dry-run flags. |

Response: `StartInstanceResponse`

| Field | Type | Description |
| --- | --- | --- |
| `stack_name` | `string` | Grouping key actually used by the provider. |
| `instance_id` | `string` | Instance identifier. |
| `urn` | `string` | Reserved compatibility field. The direct AWS SDK implementation leaves it empty. |

Current sample behavior:

- `region` must be supplied explicitly for `StartInstance`. The provider does not fall back to `scope.region` or a default region.
- If `market_type` is empty, it defaults to `INSTANCE_MARKET_TYPE_ON_DEMAND`.
- If `instance_name` is empty, the provider defaults it to `stack_name`.
- If `provider_config.ami` is empty, the provider resolves the latest Debian 13 AMI from Debian's public SSM parameters based on the instance architecture.
- The current implementation validates that `stack_name` and `instance_type` are non-empty.
- AWS-specific launch attributes such as `ami`, `subnet_id`, `security_group_ids`, `key_name`, and the default-network toggles now live under `provider_config`.

## 5. Host Integration Examples

### 5.1 Process Startup and Connection

The following pseudocode shows how a host process can launch the provider:

```go
ctx := context.Background()

process, err := runtime.Launch(ctx, "/path/to/provider-binary", os.Stderr)
if err != nil {
    return err
}
defer process.Stop()

startup := process.StartupInfo()

client, err := grpcclient.Dial(startup.Address, startup.Token)
if err != nil {
    return err
}
defer client.Close()

metadata, err := client.GetProviderInfo(ctx)
if err != nil {
    return err
}

_ = metadata
```

Key points:

- `runtime.Launch` reads the first JSON line from provider `stdout`.
- `grpcclient.Dial` establishes the gRPC connection.
- The auth token is carried in gRPC metadata rather than in business request payloads.

## 6. Integration Guidance

### 6.1 For Host Applications

- Do not tightly couple behavior to provider name. Prefer capability discovery through `GetProviderInfo`.
- Surface `warnings` in logs and UI, but do not treat them as hard failures by default.
- Add fallback handling around `ValidateConnection`, because some providers may only do structural validation or may not fully implement it yet.
- Populate `RequestContext` consistently for easier debugging and correlation.

### 6.2 For Provider Implementers

- Keep the startup JSON on `stdout` stable and send all other logs to `stderr`.
- Model stable capabilities as explicit RPCs, and keep provider-specific differences in `options` or `attributes` where practical.
- Document defaults clearly, especially region fallback, price filter defaults, and warning codes for partial matches.
- If you add a new cloud or auth scheme, prefer extending the proto instead of reusing semantically mismatched fields.

## 7. Known State of the Current Sample Provider

To set expectations for the example implementation in this repository:

- `GetProviderInfo` is implemented and returns provider metadata, auth schemes, and capability declarations.
- `Ping` is implemented and returns `pong:<payload>` with the current UTC time.
- `GetProviderSchema` is implemented and returns provider-defined resource attributes for `compute_instance`.
- `ValidateConnection` is implemented and validates STS caller identity plus regional EC2 read access.
- `ListRegions` and `ListAvailabilityZones` are implemented and expose live account location discovery.
- `ListInstanceTypes`, `GetInstanceTypeInfo`, and `GetInstancePrices` are implemented and suitable for catalog and pricing flows.
- `StartInstance` and `StopInstance` are implemented. AWS-specific launch attributes are carried through provider-defined `provider_config` instead of the shared request surface.

If this contract evolves into a fully cloud-neutral provider protocol, these areas are good candidates for further neutralization:

- Credential oneof names and enums, so vendor names are not embedded in the base protocol layer.
- Resource schemas and CRUD surfaces, so more provider-specific state can be discovered dynamically rather than modeled in the shared proto.
